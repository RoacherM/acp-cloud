# Phase 1.5: Runtime-Owned, ID-Centric Session Architecture

> **Prerequisite:** Phase 1 complete (21 commits, 50 tests passing).
> **Goal:** Deliver the correct structural skeleton Phase 2 depends on, without delivering Phase 2 features.
> **For agentic workers:** Use superpowers:subagent-driven-development or superpowers:executing-plans to implement this spec.

---

## 1. Problem Statement

Phase 1 succeeded as a technical validation spike — it proved the ACP stdio integration works end-to-end. However, its public API over-committed: types expose 9 session states when only 5 are reachable, config accepts TTL/queue/recovery parameters that are never enforced, and internal objects (`Session`, `Run`, `AgentPool`) leak into the public surface.

The result is a codebase where:

- **Session existence is tied to live objects.** Both `CloudRuntime.sessions` and `examples/server.ts` keep `Map<string, Session>`. Sleeping/crashed sessions would be unreachable.
- **Control plane is config-only.** `maxAgentProcesses`, `maxActiveSessions`, `sessionTTL`, `sleepTTL`, `maxQueueDepth` are stored but never enforced.
- **Events are run-scoped.** `RunEvent` only covers ACP `session/update`. No session-level lifecycle events, no permission delegation events.
- **Crash supervision doesn't exist.** `AgentPool` increments a counter on exit but never notifies Runtime or Session.
- **Client handler is advertised but not wired.** `AgentPool` tells agents it supports `fs.readTextFile/writeTextFile` and `terminal`, but `SandboxedFsHandler` isn't connected to the request pipeline.
- **Store owns lifecycle decisions it shouldn't.** `SessionStore.reapStale()` directly mutates session status.

Phase 1.5 fixes these structural issues so Phase 2 can layer features (persistence, recovery, sleeping, queue, permission delegation) onto a stable foundation.

---

## 2. First Principles

These principles drive every design decision below:

1. **One fact, one authority.** A session's lifecycle is owned by exactly one component (SessionController). No other component mutates session status.
2. **Persistent state vs ephemeral state must be separate.** `SessionRecord` (durable, survives restart) vs `SessionExecution` (in-memory, dies with process).
3. **Public API exposes stable resources, not process-internal object references.** Return DTOs (`SessionInfo`, `RunInfo`), not live objects.
4. **Store is passive.** CRUD + list only. No lifecycle decisions.
5. **Transport is thin.** HTTP/SSE does protocol conversion, not business logic.
6. **State machine expresses client semantics.** States tell clients "what you can do now", not implementation details.
7. **Session event stream is primary.** Run stream is a derived, filtered view.

## 3. Target Architecture

```
CloudRuntime (public API, single entry point)
├── SessionRegistry
│   ├── SessionController(sessionId="abc")
│   │   ├── SessionRecord          durable state (persisted via store)
│   │   ├── SessionExecution       ephemeral state (handle, pid, active run, timers)
│   │   ├── EventHub               session-level event stream
│   │   └── (Phase 2: RunScheduler, PermissionBroker)
│   ├── SessionController(sessionId="def")
│   │   └── ...
│   └── ...
├── SessionStore                   passive CRUD (MemorySessionStore, later FileSessionStore)
├── AgentPool                      spawn/kill + exit callback to runtime
└── ClientHandler                  fs/terminal ops, wired into ACP request pipeline
```

### Responsibility Boundaries

| Component | Owns | Does NOT Own |
|---|---|---|
| **CloudRuntime** | Public id-centric API, session registry, global admission control | Transport, auth, fan-out |
| **SessionController** | Per-session lifecycle, state transitions, event emission | Persistence mechanics, process management |
| **SessionStore** | CRUD + list on SessionRecord | Lifecycle decisions (no reapStale) |
| **AgentPool** | Process spawn/kill, ACP connection setup | Session state, recovery logic |
| **ClientHandler** | File/terminal operations within sandbox | Session lifecycle, permissions policy |
| **Server (Layer 3)** | HTTP/SSE, CORS, auth, SSE fan-out | Session state, caching Session objects |

## 5. State Machine

### 5.1 Public States

```typescript
export type SessionStatus =
  | 'initializing'  // agent spawning + ACP handshake
  | 'ready'         // idle, accepts prompts
  | 'busy'          // prompt executing, rejects new prompts
  | 'terminated';   // closed or failed, terminal state
  // Phase 2 additions:
  // | 'sleeping'    // process released, record persisted
  // | 'recovering'  // respawning after crash or wake
```

**Phase 1.5 state transitions:**

```
            ┌──────────────┐
            │ initializing │
            └──────┬───────┘
                   │ ACP handshake complete
                   ▼
            ┌──────────────┐
     ┌─────>│    ready      │<─────┐
     │      └──────┬───────┘      │
     │             │ prompt()     │ run completes
     │             ▼              │
     │      ┌──────────────┐      │
     │      │    busy       │─────┘
     │      └──────┬───────┘
     │             │ agent process dies (crash)
     │             ▼
     │      ┌──────────────┐
     └─────>│  terminated   │
            └──────────────┘
              ▲ close() from any state
```

### 5.2 Transition Rules

| From | To | Trigger | Who |
|---|---|---|---|
| (none) | initializing | `createSession()` | CloudRuntime |
| initializing | ready | ACP init + newSession complete | SessionController |
| initializing | terminated | spawn/handshake failure | SessionController |
| ready | busy | `promptSession()` | SessionController |
| busy | ready | run completes (stopReason received) | SessionController |
| busy | terminated | agent crash (exit callback) | SessionController via AgentPool callback |
| ready | terminated | `closeSession()` or agent crash | SessionController |
| * | terminated | `closeSession()` | SessionController |

### 5.3 What Each State Allows

| State | promptSession | closeSession | subscribeSession |
|---|---|---|---|
| initializing | reject | allowed | allowed (will receive status events) |
| ready | allowed | allowed | allowed |
| busy | reject (409) | allowed (kills run) | allowed |
| terminated | reject | no-op | returns empty iterable (immediately done) |

### 7.1 Session Events (Primary Stream)

```typescript
export type SessionEvent =
  // ACP session/update events (forwarded from agent)
  | AgentMessageChunkEvent

  | ToolCallUpdateEvent

  | ConfigOptionUpdateEvent
  | SessionInfoUpdateEvent
  | UsageUpdateEvent
  // Run lifecycle events (emitted by SessionController)
  | RunStartedEvent
  | RunCompletedEvent
  // Session lifecycle events (emitted by SessionController)
  | SessionStatusChangedEvent;
  // Phase 2 additions:
  // | PermissionRequestEvent
  // | SessionCrashedEvent
```

### 7.2 New Event Types

```typescript
export interface RunStartedEvent {
  type: 'run_started';
  sessionId: string;
  runId: string;
}

export interface RunCompletedEvent {
  type: 'run_completed';
  sessionId: string;
  runId: string;
  stopReason: StopReason;
}

export interface SessionStatusChangedEvent {
  type: 'session_status_changed';
  sessionId: string;
  from: SessionStatus;
  to: SessionStatus;
  reason?: string;  // e.g. 'agent_crashed', 'user_closed', 'init_failed'
}
```

### 7.3 Run-Level View (Derived)

`subscribeSession()` returns all session events. If a consumer only cares about a specific run, they filter by `runId`. This is NOT a separate stream — it's a client-side filter on the session stream.

The `Run` class (if retained internally) becomes a thin facade that:
1. Holds the `runId`
2. Provides `AsyncIterable<SessionEvent>` filtered to events for this run
3. Has no state of its own — all state lives in `SessionController`

## 9. AgentPool Changes

### 9.1 Exit Supervision Callback

```typescript
export interface AgentPoolConfig {
  agents: Record<string, AgentDefinition>;
  onProcessExit?: (handle: AgentHandle, code: number | null, signal: string | null) => void;
}
```

`AgentPool` calls `onProcessExit` on every child exit. CloudRuntime wires this to find the owning SessionController and trigger the appropriate state transition (busy/ready → terminated, with reason `'agent_crashed'`).

### 9.2 Client Handler Integration

Phase 1 declared `fs.readTextFile`, `fs.writeTextFile`, and `terminal` capabilities during ACP `initialize`. Phase 1.5 must either:

**Option A (recommended):** Wire `SandboxedFsHandler` into the `Client` object that `AgentPool` creates, so `readTextFile` and `writeTextFile` requests from the agent are actually handled.

**Option B:** Remove the capability declaration from `initialize` (set `fs: {}` and `terminal: false`). This is honest but may break agents that expect filesystem access.

**Decision: Option A.** Wire it in. The handler already exists and is tested. The `Client` interface in `AgentPool.spawn()` already has the right shape — it just needs to dispatch `readTextFile`/`writeTextFile` to `SandboxedFsHandler`.

To provide the session's `cwd` to the handler, `AgentPool.spawn()` gains an optional `cwd` parameter. SessionController passes its `cwd` when spawning. AgentPool creates a `SandboxedFsHandler(cwd)` and wires it into the `Client` object.

`terminal` capability should be set to `false` until a real terminal handler exists.

## 10. Admission Control (Phase 1.5 Scope)

CloudRuntime enforces two limits at session creation time:

```typescript
async createSession(opts: CreateSessionOptions): Promise<SessionInfo> {
  // Check maxActiveSessions
  const activeSessions = await this.store.list();
  const nonTerminated = activeSessions.filter(s => s.status !== 'terminated');
  if (nonTerminated.length >= this.config.maxActiveSessions) {
    throw new Error('Max active sessions reached');
  }

  // Check maxAgentProcesses
  if (this.pool.stats().active >= this.config.maxAgentProcesses) {
    throw new Error('Max agent processes reached');
  }

  // Proceed with creation...
}
```

Phase 1.5 does NOT implement eviction (freeing slots by sleeping/terminating idle sessions). It simply rejects when at capacity. Eviction is Phase 2.

## 13. Success Criteria

Phase 1.5 is complete when:

1. **Public API is id-centric.** All session operations go through `CloudRuntime` methods that take `sessionId`. No `Session` or `Run` objects in public exports.
2. **State machine is honest.** Only 4 states have behavior (`initializing`, `ready`, `busy`, `terminated`). No unreachable states in the type.
3. **Events are session-level.** `SessionEvent` stream includes `run_started`, `run_completed`, `session_status_changed` alongside ACP events.
4. **Admission control works.** `createSession` rejects when `maxAgentProcesses` or `maxActiveSessions` exceeded.
5. **Crash supervision works.** Agent process dying triggers `terminated` status + `session_status_changed` event.
6. **Client handler is wired.** Agent `readTextFile`/`writeTextFile` requests route to `SandboxedFsHandler`. Terminal capability is not declared.
7. **Store is passive.** No `reapStale` on `SessionStore` interface.
8. **All tests pass.** Updated tests cover new API surface.
9. **Examples work.** `server.ts` and `basic-usage.ts` use id-centric API.
10. **Pi agent still works.** End-to-end validation with real agent unchanged.
